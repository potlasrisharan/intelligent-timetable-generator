1. Executive Summary
The Intelligent Timetable Generator is a full-stack, constraint-based academic scheduling system designed to automatically generate conflict-free timetables for university environments. The system replaces error-prone, manual spreadsheet scheduling with a formal constraint programming engine (Google OR-Tools). This engine acts as the "brain" of the application, mathematically guaranteeing schedule correctness while optimizing for quality metrics such as faculty workload balance, room utilization, and student timetable compactness. The final product includes a comprehensive admin dashboard, conflict resolution tools, and beautiful PDF/Excel exports.
2. Problem Statement
Educational institutions face an incredibly complex puzzle when creating schedules. The system must simultaneously balance:
	•	Multiple sections existing for the same semester and department.
	•	Theory vs. Labs: Courses require separate theory and practical (lab) hours.
	•	Combined Sections: Sometimes two or more sections are merged into one large room with one teacher.
	•	Faculty Availability: Teachers have specific days they are available and a maximum number of classes they can teach per day.
	•	Room Capacity: A class of 60 students cannot be put into a room that only holds 40. Labs require specialized lab rooms.
	•	Lunch Breaks & Hours: Fixed institutional working hours and mandatory lunch breaks.
	•	Consecutive Labs: Lab sessions must be scheduled back-to-back on the same day.
	•	No Overlaps: A teacher or a room cannot be in two places at once.
Manual scheduling at this scale takes 40+ hours per semester, often resulting in undetected clashes and suboptimal use of time. A formal constraint-aware automated system is the only reliable solution.
3. System Modules Overview
To make this giant system manageable, it is divided into five core building blocks that interact seamlessly:
	•	Input Configuration Module: The data entry point. This is where admins define the basic rules of the puzzle—courses, sections, teachers, and rooms.
	•	Constraint Engine (Scheduling Core): The mathematical solver. It takes the data from the Input Module, applies strict rules (constraints), and calculates the perfect timetable.
	•	Conflict Detection & Resolution: The safety net. If a human admin tries to manually change the schedule, this module checks if the new change breaks any rules and suggests better alternatives.
	•	Output Generation: The presentation layer. It takes the mathematical result and turns it into readable, color-coded PDF and Excel grids.
	•	Admin Dashboard: The central control room. It provides a visual interface for the admin to trigger generation, view health metrics, and manage the other four modules.

6. Authentication, Authorization & Security
To ensure data integrity, the system uses strict security protocols:
	•	Authentication: JWT-based stateless authentication (Access token: 1 hour TTL, Refresh token: 7 days TTL).
	•	Password Security: Passwords hashed using bcrypt (minimum cost factor 12).
	•	Password Policy: Minimum 8 characters, 1 uppercase, 1 number, 1 special character.
	•	Rate Limiting: Maximum 5 failed login attempts per minute per IP before a 15-minute lockout.
	•	Encryption: HTTPS required for all transit.
	•	Role-Based Access Control (RBAC): Users are strictly limited to the permissions outlined in Section 5. API endpoints validate roles on every request.
	•	Audit Logs: Every write, delete, and generation action is logged immutably.
7. System Workflow / User Workflow
Step-by-step administrative journey from login to published timetable:
	•	Authentication: Admin logs in and is routed to the Dashboard.
	•	Data Ingestion: Admin uploads CSVs for Courses, Faculty, Rooms, and Sections (or uses the UI to input data manually).
	•	Configuration: Admin defines timeslots, lunch periods, holidays, and combined sections.
	•	Generation: Admin clicks "Generate Timetable". The system sends an async request to the Celery worker and displays a loading progress bar.
	•	Review & Adjust: Admin reviews the DRAFT timetable in the Timetable Editor. They use the Conflict Panel to resolve any minor issues or manual adjustments.
	•	Activation: Admin clicks "Publish/Activate". The DRAFT becomes ACTIVE, and the previous ACTIVE schedule is ARCHIVED.
	•	Export & Distribution: Admin exports the ACTIVE timetable to PDF/Excel for physical printing or external distribution.

10. Constraint Engine — Formal Specification
This is the core mathematical model powered by Google OR-Tools CP-SAT.
10.1 Decision Variables

10.2 Hard Constraints (Must NEVER be violated)
	•	HC-01: No faculty double-booking.
	•	HC-02: No room double-booking.
	•	HC-03: No section double-booking.
	•	HC-04: Room capacity enforcement (room.capacity >= section.student_count).
	•	HC-05: Lab sessions MUST be in LAB-type rooms.
	•	HC-06: Theory sessions MUST be in CLASSROOM-type rooms.
	•	HC-07: The Lunch slot is always empty.
	•	HC-08: Faculty availability windows are strictly respected.
	•	HC-09: Faculty max_periods_per_day is not exceeded.
	•	HC-10: Required weekly hours (theory + practical) are exactly fulfilled.
	•	HC-11: Practical sessions (Labs) must be consecutive on the same day.
	•	HC-12: Combined sections share the exact same slot, room, and faculty.
	•	HC-13: Holiday slots are completely blocked.
	•	HC-14: "Locked" slots (manually fixed by admins) survive regeneration unchanged.
	•	HC-15: Department-exclusive rooms are only assigned to their respective departments.
	•	HC-16: Scheduling only occurs within defined working days/hours.
10.3 Soft Constraints & Objective Function
The solver tries to optimize the schedule by minimizing a penalty score:
	•	SC-01 (Weight 10): Minimize faculty idle gaps.
	•	SC-02 (Weight 8): Balance workload across days.
	•	SC-03 (Weight 6): Prefer faculty preferred slots.
	•	SC-04 (Weight 4): Avoid last period classes.
	•	SC-05 (Weight 5): Spread subjects across the week.
	•	SC-06 (Weight 3): Minimize room changes for sections.
	•	SC-07 (Weight 2): Balance Morning/Afternoon load.
	•	SC-08 (Weight -7): Maximize room utilization.
	•	Objective Function: Minimize . Time limit: 25 seconds.
10.4 Solver Failure (UNSAT) Handling & Relaxation Wizard
If the puzzle is mathematically impossible:
	•	The solver returns INFEASIBLE.
	•	A Conflict Analyzer runs to find the Irreducible Infeasible Subset (IIS).
	•	The system displays a human-readable error.
	•	A Relaxation Wizard UI pops up, suggesting the easiest fixes.
11. Conflict Detection & Resolution
11.1 Conflict Types & Auto-Suggestions
	•	Double Booking / Capacity Break: Critical severity. The system suggests the next available free room/slot.
	•	Faculty Overload: High severity. System suggests moving the class to a lighter day.
11.2 Manual Override Rules
	•	Hard Constraint Override: BLOCKED. UI rejects drops causing mathematical impossibilities.
	•	Soft Constraint Override: ALLOWED. UI accepts drops but logs a warning about Quality Score degradation.
11.3 Resolution Workflow
	•	Conflict detected during manual edit.
	•	Conflict Panel opens showing the exact violated constraint.
	•	System lists Top 3 alternative slots ranked by lowest penalty score.
	•	Admin clicks a suggestion to auto-resolve.
12. Timetable Versioning & Regeneration
12.1 Timetable Version Lifecycle State Machine
Timetables exist as distinct snapshots passing through a strict lifecycle:
	•	DRAFT: The initial state of a newly generated or modified timetable. Only visible to admins.
	•	Transition -> ACTIVE: When admin clicks "Publish".
	•	ACTIVE: The live, published schedule visible to faculty and students. (Only ONE Active version allowed per department).
	•	Transition -> ARCHIVED: Automatically happens to the old ACTIVE version when a new DRAFT is published.
	•	ARCHIVED: Read-only historical record.
	•	Transition -> RESTORED (ACTIVE): Admin triggers a rollback, making this the ACTIVE version again.
12.2 Regeneration Modes
	•	Full Regenerate: Solves from scratch. Erases manual overrides; preserves Locked Slots.
	•	Partial Regenerate: Freezes satisfactory slots and re-solves only empty/unlocked areas.
	•	Conflict-Only Re-solve: Focuses solver exclusively on slots currently throwing errors.

19.1 Performance & Scaling Strategy
To handle multiple departments and massive datasets concurrently:
	•	Solver Isolation: Generation tasks are strictly offloaded from the main API thread to isolated Celery workers.
	•	Horizontal Scaling: As load increases, additional Docker containers running Celery workers can be spun up. Redis manages task distribution automatically.
	•	Database Optimization: Indexes placed on dept_id, version_id, and faculty_id. Connection pooling managed by SQLAlchemy to prevent connection exhaustion.
	•	Caching: Department master data (Rooms, Courses) cached in Redis for fast API retrieval.
20. Error Handling & Edge Cases
The system anticipates and gracefully handles failure states:
	•	Solver Infeasible: Triggers the Conflict Analyzer and Relaxation Wizard (Section 10.4).
	•	Solver Timeout (>60s): Aborts generation, returns the best partial solution found, and prompts the user to relax constraints.
	•	Concurrent Generation: If Admin B clicks "Generate" while Admin A's job is running for the same department, the API returns 409 Conflict: Generation already in progress.
	•	CSV Import Errors: Entire transaction is rolled back if a single row fails validation. Returns a detailed array of errors (e.g., [{"row": 12, "error": "Room capacity missing"}]).
	•	Holiday Added Post-Generation: A background cron job checks active timetables against new holidays. It flags affected slots in red on the UI with a "Re-solve Required" warning.
	•	Export Failures: If PDF generation fails due to memory limits, the API catches the exception and returns a 500 Server Error with a fallback link to the Excel export.

23. System Data Flow
To understand how data moves through the application from start to finish:
	•	Input Phase: The user enters configuration data (Courses, Faculty, Rooms) via the Frontend UI or CSV upload. The FastAPI backend validates this and stores it in PostgreSQL.
	•	Dispatch Phase: When the user clicks "Generate", the API compiles the data into a mathematical payload and dispatches a job to Redis.
	•	Execution Phase: A Celery Worker picks up the job from Redis, feeds the data to the OR-Tools CP-SAT solver, and computes the best possible timetable matrices.
	•	Storage Phase: The completed schedule is written back to PostgreSQL as a DRAFT Timetable Version, along with its calculated Quality Score.
	•	Presentation Phase: The Frontend fetches the new DRAFT data, rendering it visually in the Timetable Editor for review.
	•	Publishing & Export Phase: The Admin activates the timetable. Once ACTIVE, the reporting engine can dynamically query the database to generate utilization analytics or render binary PDF/Excel exports.
24. Entity Relationship Summary
The relational database represents the scheduling domain through heavily linked entities:
	•	Departments act as the root tenant for all data.
	•	Courses, Faculty, Sections, and Rooms are child entities of Departments.
	•	Combined Sections act as a bridging entity, linking a single Course and Faculty to multiple Sections.
	•	Timetable Versions represent discrete generation events linked to a Department.
	•	Timetable Entries are the atomic scheduled blocks. They belong to a single Version and maintain foreign keys to the assigned Course, Section, Faculty, Room, and Timeslot.

25. Constraint Priority System
The system manages conflicts by evaluating rules against a strict hierarchy of precedence:
	•	Priority Level 1 (Absolute - Locked Slots): These are human-defined, pinned slots. The solver is strictly forbidden from moving them under any circumstance.
	•	Priority Level 2 (Strict - Hard Constraints): Foundational math rules (e.g., no double-booking). The solver MUST satisfy these. If it cannot, the generation fails (UNSAT).
	•	Priority Level 3 (Manual - Admin Overrides): A human admin can bypass soft constraints (e.g., forcing a teacher into a non-preferred slot). However, the UI will still block the admin if they try to violate a Level 2 Hard Constraint.
	•	Priority Level 4 (Optimization - Soft Constraints): The solver attempts to satisfy these rules to make the timetable "nicer," but will freely break them to satisfy Level 1 or Level 2 rules.
26. Schedule Quality Score Calculation
The Quality Score is an integer metric reflecting how well the timetable satisfies human preferences. It is calculated during the generation process.
Formula: Total Penalty Score = Σ (Violations of Soft Constraint 'i' × Weight 'i')
Example: If a schedule forces a faculty member into 2 non-preferred slots (Weight: 6) and creates 1 idle gap (Weight: 10), the penalty added is (2 × 6) + (1 × 10) = 22.
A lower Quality Score indicates a more optimized, human-friendly timetable. A score of 0 means a theoretically perfect schedule. Negative scores can be achieved if bonus weights (like maximizing room utilization) outweigh penalties.
27. Security Threat Model
The application proactively defends against common web vulnerabilities:
	•	SQL Injection: Entirely mitigated by using SQLAlchemy ORM, which automatically parameterizes all database queries.
	•	Unauthorized Access / Privilege Escalation: Prevented via strict Role-Based Access Control (RBAC). Department IDs are checked on every API request to prevent cross-tenant data access.
	•	Token Misuse: JWTs are short-lived (1 hour). If a user logs out, their refresh token is immediately pushed to a Redis blocklist to prevent reuse.
	•	File Upload Attacks: CSV import endpoints validate the MIME-type, enforce a strict file size limit (e.g., 5MB), and parse data safely using Pandas before it ever touches the database.
	•	API Abuse (DDoS/Brute Force): Redis handles rate limiting at the API gateway level, throttling auth attempts to 5 per minute and general data requests to 20 per second per IP.
28. Assumptions & Limitations
System design operates under the following known boundaries:
	•	Max Scale: Optimized for up to 10 departments per tenant, and up to 20 sections per department.
	•	Time Grid: Supports a maximum of 10 periods per day.
	•	Solver Timeouts: To prevent infinite hanging, the solver enforces a strict 60-second absolute timeout. If a perfect solution isn't found, it returns the best feasible schedule up to that point.
	•	Lab Continuity: Labs must be scheduled consecutively on the same day. The system does not support labs that span across multiple days.
	•	Registration Data: The system assumes student cohorts are grouped into rigid "Sections". It does not handle individualized, a-la-carte student course registrations dynamically.
29. Glossary
	•	Hard Constraint: A mandatory rule. If broken, the timetable is completely invalid.
	•	Soft Constraint: A preference. Broken rules add a penalty to the Quality Score but the timetable is still valid.
	•	UNSAT (Infeasible): A state where the puzzle is mathematically impossible to solve with the current rules.
	•	CP-SAT: Constraint Programming Satisfiability. The specific mathematical engine within Google OR-Tools used to solve this puzzle.
	•	Combined Section: Multiple student groups sharing one teacher and one room at the same time.
	•	Locked Slot: A timetable block manually pinned by an admin that the solver is not allowed to move.
	•	Quality Score: The total sum of all soft constraint penalties. The algorithm tries to make this number as close to zero as possible.
	•	Relaxation Wizard: A tool that helps humans figure out which rules to "relax" or change when a puzzle becomes UNSAT.

	•	Regression Testing: The entire test suite executes automatically via a CI/CD pipeline upon any pull request to the main repository branch.
